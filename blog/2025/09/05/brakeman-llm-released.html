Brakeman-LLM Released

LLM-powered sprinkles for Brakeman

A long time ago, when Brakeman was first designed, its output was optimized to fit into tables like this, with one warning per line:

+------------+----------------+--------------+-------------------+--------+------------------------------------------------------------------------+
| Confidence | Class          | Method       | Warning Type      | CWE ID | Message                                                                |
+------------+----------------+--------------+-------------------+--------+------------------------------------------------------------------------+
| High       | HomeController | test_command | Command Injection | [77]   | Possible command injection near line 34: `ls #{+params[:file_name]+}`  |
| High       | HomeController | test_command | Command Injection | [77]   | Possible command injection near line 36: system(+params[:user_input]+) |
+------------+----------------+--------------+-------------------+--------+------------------------------------------------------------------------+

This severely limited the “message” that described the warning. Brakeman has largely stuck to this, even though the default output now looks like this:

Confidence: High
Category: Command Injection
Check: Execute
Message: Possible command injection
Code: `ls #{params[:file_name]}`
File: app/controllers/home_controller.rb
Line: 34

Confidence: High
Category: Command Injection
Check: Execute
Message: Possible command injection
Code: system(params[:user_input])
File: app/controllers/home_controller.rb
Line: 36

From the beginning, Brakeman was intended as tool primarily for developers, not security professionals. However, it’s pretty clear that “Possible command injection” isn’t enough content for most non-security folks.

Also, wouldn’t it be nice if the message used some of the known context about the warning? Like the code involved? And how about recommendations on how to fix the potential security issue??

Historically, this was pretty hard and I didn’t have a lot of motivation to build out something that complex.

Brakeman-LLM

The brakeman-llm gem extends Brakeman with LLM-generated warning descriptions.

Under the hood, this is actually pretty simple, other than the bits that hook into Brakeman. Once a scan is complete, each warning is sent to an LLM of your choosing with a basic prompt to explain the warning and recommendations for how to fix it.

The quality of the results naturally depend on your LLM, but here is an example using Anthropic Claude:

Confidence: High
Category: Command Injection
Check: Execute
Code: `ls #{params[:file_name]}`
File: app/controllers/home_controller.rb
Line: 34
Message: Possible command injection

This Brakeman warning indicates a critical Command Injection vulnerability in the Ruby on Rails application. The issue is located in the HomeController class, specifically in the test_command method on line 34 of the home_controller.rb file.

The vulnerable code executes a shell command using backticks (`) to run the 'ls' command with a file name supplied by the user through the params[:file_name] parameter. This is highly dangerous because it allows an attacker to inject arbitrary shell commands that will be executed on the server.

For example, an attacker could supply a file name like "; rm -rf /" which would attempt to delete all files on the system. Or they could use "& cat /etc/passwd" to read sensitive system files.

To fix this vulnerability, avoid using user input directly in shell commands. Instead, use built-in Ruby methods for file operations. For listing files, you can use the Dir.glob or Dir.entries methods:

```ruby
files = Dir.glob("#{Rails.root}/public/#{params[:file_name]}*")
```

If you absolutely must use external commands, ensure you properly sanitize and validate all user input. You can use Shellwords.escape to escape special characters:

```ruby
require 'shellwords'
safe_filename = Shellwords.escape(params[:file_name])
output = `ls #{safe_filename}`
```

Additionally, consider using Ruby's File class methods instead of shell commands when possible. For example:

```ruby
File.exist?(params[:file_name])
File.directory?(params[:file_name])
```

Always validate and sanitize user input, use the principle of least privilege, and avoid using shell commands with user-supplied data whenever possible. Implement proper input validation and consider using a whitelist of allowed file names or directories to further enhance security.

(The above message is auto-generated and may contain errors.)

Note that the description above not only shows how to avoid command injection, it provides better approaches to listing files generally (which is what the original code was doing).
